# 开发日志 - 阶段04：数据基础设施 — 用户行为采集与特征工程管道

## 1. 阶段目标与完成情况

本阶段聚焦在 **FR1（多维用户行为数据模拟与采集）** 的完整落地，为后续商务智能预测层提供数据支撑。

### 已完成的核心工作

- [x] **数据库扩展**：在 `db/schema.sql` 新增：
  - `user_behavior_event`：用户行为事件表（支持 CLICK/FAVORITE/ADD_TO_CART/BROWSE/SHARE/PURCHASE 六种行为，含浏览时长 dwell_seconds、扩展属性 JSONB 字段）
  - `user_feature_matrix`：用户特征矩阵表（12 维基础特征 + 意愿得分 + 4 个索引）
- [x] **Java 端行为采集 API**：
  - `UserBehaviorEvent` 实体 + `UserBehaviorEventMapper`（含批量插入、按用户/行为查询、近期行为计数）
  - `UserFeatureMatrix` 实体 + `UserFeatureMatrixMapper`（含 upsert、意愿得分更新）
  - `UserBehaviorService` / `UserBehaviorServiceImpl`：完整的特征工程管道，含时间衰减加权、行为分类统计、近7天/1天活跃度计算
  - `UserBehaviorController`：`POST /api/behavior/event`（单条采集）、`POST /api/behavior/events/batch`（批量采集）、`GET /api/behavior/events`、`POST /api/behavior/feature-matrix/compute`、`GET /api/behavior/feature-matrix`
- [x] **goods-app 编译配置更新**：`pom.xml` includes 中新增行为采集相关 7 个类
- [x] **网关白名单更新**：`/api/behavior/event` 和 `/api/behavior/events/batch` 加入免鉴权白名单
- [x] **前端行为埋点 SDK**：`frontend/src/api/behavior.ts`
  - 自动队列化批量上报（3秒/20条阈值）
  - 便捷方法：`trackClick`/`trackFavorite`/`trackAddToCart`/`trackBrowse`/`trackShare`/`trackPurchase`
  - 浏览时长计时器：`startDwellTimer`/`stopDwellTimer`
  - 页面卸载前自动 flush
- [x] **Python BI 模块基础架构**：`bi/` 目录
  - `app/config.py`：统一配置（DB/Redis/模型路径/阈值）
  - `app/pipeline/data_loader.py`：SQLAlchemy 数据加载器（行为事件、特征矩阵、带标签数据）
  - `app/pipeline/feature_engine.py`：特征工程（12 基础特征 + 5 派生特征：interaction_intensity/purchase_ratio/cart_to_browse_ratio/recent_activity_ratio/dwell_per_browse）
  - `data/generate_mock_data.py`：模拟数据生成（500 用户 × 50000 事件，高意愿用户偏置）
  - Dockerfile + docker-compose 集成

## 4. 给后来者的指导

- 行为事件写入是高频操作，生产环境建议先写 Redis Stream 再异步落库，当前版本直写 PG 适合课程演示
- 特征矩阵的 `computeFeatureMatrix` 是重计算，应在离线管道中定期调用，不要在在线请求中同步计算
- Python BI 模块与 Java 微服务仅通过 PostgreSQL 和 Redis 进行状态同步（NFR4 架构解耦）
